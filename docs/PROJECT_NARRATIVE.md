# KONAMOBILITY V2 — 콜카드 × 기사 성향 매칭 스튜디오
### 배차 실패(Expired) 감소를 위한 22차원 벡터 매칭 프로젝트 내러티브

> 이 문서는 **PPT 발표 자료로 변환**하는 것을 전제로 작성되었습니다.
> 각 섹션이 곧 슬라이드 1~N장에 대응하며, 비전공자(경영/사업팀)도 이해할 수 있도록
> 핵심 개념(특히 코사인 유사도)을 단계적으로 풀어서 설명합니다.
> 기술 세부는 각 섹션 하단의 `▸ 기술 부록`에 분리해 두었습니다.

## 슬라이드 0 — 한 줄 요약 (Executive Summary)

> **"이 콜카드는 어떤 기사에게 보내야 잡힐까?"** 를
> 콜카드와 기사의 운행 성향을 **22개 숫자(벡터)** 로 바꿔
> **방향이 가장 비슷한 기사**를 찾아주는 매칭 엔진을 만들었고,
> 이를 직접 눈으로 검증하는 **매칭 스튜디오 시뮬레이터**까지 구축했습니다.
> 최종 목표는 이 매칭 결과를 배차 로직에 연결해 **Expired(미배차)를 줄이는 것**입니다.

# 1. 왜 시작했는가 — 문제 정의 (Why)

## 1.1 핵심 문제: 콜이 잡히지 않고 사라진다 (Expired)

택시 호출(콜카드)이 들어와도 **아무 기사도 수락하지 않으면 콜은 "Expired" 상태로 소멸**합니다.
- 승객 → 대기 후 이탈 (이용 경험 악화)
- 기사 → 빈 차 운행 (수익 손실)
- 플랫폼 → 거래 실패 (GMV 손실)

## 1.2 관찰: Expired에는 "패턴"이 있었다

단순히 기사가 부족해서 생기는 문제가 아니었습니다.
**특정 요일·시간대·장소 조합에서 Expired가 반복적으로** 발생했습니다.

| 관찰된 패턴 축 | 예시 |
|---|---|
| **시간대** | 새벽 시간대 장거리 콜이 유독 잘 안 잡힘 |
| **요일** | 주말 심야 vs 평일 출근 시간의 수락률 차이 |
| **장소** | 외곽/특정 행정동(ASP)에서 출발하는 콜의 실패 집중 |
| **콜 속성** | 거리·요금·유료/무료·탄력/일반 조합별 선호 편차 |

→ **"기사마다 좋아하는 콜의 성향이 다르다"** 는 가설.
즉, 같은 콜이라도 **그 콜의 성향에 맞는 기사**에게 보내면 수락 확률이 올라간다.

## 1.3 그래서 던진 질문

> **"콜카드의 성향과 기사의 운행 성향을 수치화해서,
> 가장 궁합이 맞는 기사에게 우선 배차할 수 없을까?"**

이것이 프로젝트의 출발점입니다. 사람의 직관("이 기사는 야간 장거리를 잘 받지")을
**데이터로 정량화**하는 것이 목표였습니다.

# 2. 어떻게 접근했는가 — 22차원 벡터 + 코사인 유사도 (How)

## 2.1 핵심 아이디어: "성향을 좌표로 바꾼다"

콜카드와 기사를 **같은 좌표계 위의 점(벡터)** 으로 표현하면,
**서로 얼마나 비슷한 방향을 가리키는지**로 궁합을 계산할 수 있습니다.

> 비유: MBTI를 숫자로 바꾼 것과 같습니다.
> "이 콜은 야간·장거리·고요금 성향(콜의 MBTI)"
> "이 기사는 야간·장거리를 선호(기사의 MBTI)" → **성향이 같은 방향이면 매칭**

## 2.2 22개의 축(차원) — 무엇으로 성향을 표현하는가

콜카드와 기사를 **22개 항목으로 점수화**합니다. (각 항목 = 1개 차원/축)

| 그룹 | 차원 | 개수 |
|---|---|---|
| **시간대** | 새벽 / 오전 / 주간 / 야간 | 4 |
| **요일** | 월 / 화 / 수 / 목 / 금 / 토 / 일 | 7 |
| **거리** | 단거리(≤3km) / 중거리(3~8km) / 장거리(>8km) | 3 |
| **요금** | 저요금(≤1만) / 중요금(1~2만) / 고요금(>2만) | 3 |
| **콜 유형** | 유료콜 / 무료콜 | 2 |
| **상품** | 탄력 / 일반 | 2 |
| **ETA(근접성)** | 근접성 — *매칭 점수에서는 제외* | 1 |
| | **합계** | **22** |

### 콜카드는 "원-핫(one-hot)", 기사는 "선호 점수"

- **콜카드**: 그 콜이 가진 속성만 `1`, 나머지는 `0`.
  예) *금요일 야간 장거리 고요금 일반콜* →
  `야간=1, 금=1, 장거리=1, 고요금=1, 일반=1`, 나머지 0.

- **기사**: 과거 운행 이력에서 학습한 **선호 강도(0~1 연속값)**.
  예) 야간을 자주 받는 기사 → `score_night ≈ 0.8`, 새벽 거의 안 받음 → `score_dawn ≈ 0.1`.

> 즉 **콜 = "나는 이런 콜이다"**, **기사 = "나는 이런 콜을 좋아한다"**.
> 두 벡터의 방향이 일치할수록 궁합이 좋습니다.

## 2.3 코사인 유사도 — "방향이 얼마나 같은가"

두 벡터가 얼마나 비슷한지를 재는 방법으로 **코사인 유사도**를 사용합니다.
**두 벡터 사이의 각도**를 봅니다.

```
                 A · B            (두 벡터를 곱해 더한 값)
cos θ  =  ─────────────────
              ‖A‖ × ‖B‖        (각 벡터의 길이의 곱)
```

- **각도가 0°에 가까움 → cos θ ≈ 1 → 성향이 거의 동일 (찰떡궁합)**
- 각도가 90° → cos θ = 0 → 성향이 전혀 무관
- 값의 범위: 0 ~ 1 (성향 점수는 음수가 없으므로)

### 왜 "거리"가 아니라 "각도(방향)"인가?

기사마다 **활동량(콜 수)이 다릅니다.** 활동량이 많은 기사는 모든 점수가 크고,
적은 기사는 작습니다. 단순 거리로 비교하면 **활동량 많은 기사가 무조건 유리**해집니다.
코사인은 **크기를 정규화(‖A‖×‖B‖로 나눔)** 하므로,
"얼마나 많이 뛰었나"가 아니라 **"무엇을 선호하나(방향)"** 만 순수하게 비교합니다.

### 비전공자용 직관

> 화살표 두 개를 겹쳐 놓았을 때 **벌어진 각도**를 보는 것입니다.
> 두 화살표가 같은 쪽을 가리키면(각도 작음) "성향이 같다 = 점수 1점에 가까움",
> 서로 직각이면 "관련 없음 = 0점".
> 화살표의 **길이(=얼마나 바쁜 기사인지)는 무시**하고 **방향만** 봅니다.

### 간단 계산 예시

- 콜: `[야간1, 금1, 장거리1]` (나머지 0)
- 기사 A: `[야간0.9, 금0.8, 장거리0.7]` → 같은 축에 높은 점수 → **cos θ ≈ 0.95 (높음)**
- 기사 B: `[새벽0.9, 월0.8, 단거리0.7]` → 콜이 가진 축은 전부 0 → **cos θ = 0 (무관)**

→ 같은 콜인데 A에게 보내면 잡힐 확률이 높다는 것을 **수치로** 보여줍니다.

## 2.4 공간(H3) 점수 — "이 동네를 좋아하는 기사인가"

성향(시간·요일·거리·요금)만으로는 부족합니다. **지리적 궁합**도 중요합니다.
지구를 육각형 격자로 나누는 **Uber H3(해상도 7)** 를 사용합니다.

- 각 기사의 **선호 출발지/도착지 구역**(과거·선호 이력 기반)을 육각형으로 보유.
- 콜의 출발/도착 육각형이 기사의 선호 구역과 **얼마나 가까운지**(격자 거리)로 점수화.
- 공간 점수 = `출발지 궁합 × 45% + 도착지 궁합 × 55%` (도착지에 더 가중).

> ⚠️ **중요: H3는 기사의 실시간 GPS 위치가 아닙니다.**
> "이 기사가 평소 **선호하는/자주 다니는** 출발·도착 구역"이라는
> **이력·선호 데이터**입니다. (실시간 위치 추적이 아님)

## 2.5 최종 매칭 점수 — 성향 75% + 공간 25%

```
최종 점수 = (코사인 성향 점수 × 0.75) + (H3 공간 점수 × 0.25)
```

- **성향(코사인) 75%** : 시간·요일·거리·요금·콜유형·상품 궁합
- **공간(H3) 25%** : 지리적 선호 구역 궁합

### ⚠️ ETA는 최종 점수에 들어가지 않는다

22개 차원 중 **ETA(근접성)** 는 벡터에 저장·화면 표시는 하지만
**코사인 매칭 계산에서는 제외**합니다.
- 이유: ETA(도착 예상 시간)는 **실시간 교통 상황**에 따라 출렁이므로,
  "성향 궁합"이라는 안정적 신호를 흔들면 안 됩니다.
- 따라서 매칭에 실제로 쓰이는 성향 차원은 **21개**, ETA는 **참고용 표시 정보**입니다.

> ▸ 기술 부록 (2장)
> - 벡터 정의/변환: `lib/matching-vector.ts`
>   - `callToVector()` (콜 원-핫), `driverToVector()` (기사 선호 점수)
>   - `cosineSimilarityForMatching()` — ETA 차원(`score_near`) 제외 후 코사인
>   - `MATCHING_EXCLUDED_DIMENSION_KEYS` = `['score_near']`
> - 공간 점수: `lib/h3-match-score.ts`
>   - `V2_MATCH_WEIGHTS = { similarity:0.75, spatial:0.25, originWithinSpatial:0.45, destinationWithinSpatial:0.55 }`
>   - `calculateV2FinalScore(similarity, spatial)`
> - 회귀 테스트: `lib/__tests__/matching-eta-exclusion.test.ts`
>   ("ETA만 바뀌어도 finalScore/similarityScore 불변" 보장)

# 3. 무엇을 만들었는가 — 매칭 스튜디오 시뮬레이터 (What)

## 3.1 목적: "매칭이 맞는지 눈으로 검증하는 실험실"

배차 로직에 바로 넣기 전에, **매칭 결과가 타당한지 사람이 검증**할 수 있어야 합니다.
그래서 만든 것이 **매칭 스튜디오**(`/matching-studio`) — 풀 화면 Google 지도 위에
모든 정보를 유리(glass) 카드로 띄우는 인터랙티브 대시보드입니다.

## 3.2 두 가지 입력 모드

| 모드 | 설명 | 용도 |
|---|---|---|
| **콜카드 기준** | 실제 DB의 콜카드 하나를 선택 | 실데이터로 "이 콜은 누구에게?" 검증 |
| **시나리오 입력** | 출발지·도착지를 직접 검색해 가상 콜 생성 | "만약 이런 콜이라면?" 가설 실험 |

- 상단 데이터 필터(ASP 행정동 / 날짜 / 시간 / 상태)로 콜카드 풀을 좁힘.
- 입력만 하면 **자동으로** Top 10 후보 기사가 계산됨 (별도 실행 버튼 없음).

## 3.3 화면에서 보여주는 것

1. **Top 10 후보 기사** (우측 패널) — 최종 점수 순 랭킹.
2. **KPI 상단바** — 선택 콜카드 / 후보 기사 수 / 1위 점수 / 성향 점수 / 공간 점수.
3. **지도 시각화**
   - 출발지·도착지 마커 (붉은 점, 맥동 애니메이션)
   - 승객 이동 경로 **트레일 이펙트**(혜성처럼 흐르는 붉은 경로) + 거리·시간(ETA)
   - 선택 기사의 **선호 H3 육각형**(출발=청록, 도착=보라)
4. **추천 근거(Evidence) 드로어** (하단) — 왜 이 기사가 추천됐는지 3분할로 분해:
   - **점수 분해**: 코사인 75% + 공간 25% 기여도
   - **벡터 근거**: 22개 차원 중 어떤 축이 일치했는지
   - **공간 근거**: 출발/도착 H3 궁합 거리

## 3.4 기술 스택 (구현 메모)

- **지도 타일/POI 검색**: Google Maps JS API (클라이언트)
- **경로/ETA**: **TMap API** (한국 국내 경로 — Google Routes는 국내 미지원)
- **데이터 레이어**: deck.gl (마커/경로/H3 육각형 렌더링)
- **프레임워크**: Next.js (App Router)

> ▸ 기술 부록 (3장)
> - 페이지: `app/components/v2/matching/MatchingStudio.tsx`
> - 상태/자동실행 로직: `useMatchingStudio.ts`
> - 지도/애니메이션: `useGoogleMap.ts`
> - 경로(TMap): `lib/server/tmap-routes.ts`, `app/api/maps/routes/route.ts`
> - 매칭 모델: `lib/matching-studio-model.ts`, `lib/adapters/matching.ts`

# 4. 그래서 무엇을 얻는가 — 배차 로직 연결 & Expired 감소 (Result / Goal)

## 4.1 매칭 → 배차 로직으로의 전달

매칭 스튜디오로 **검증된 매칭 로직**을, 실제 **배차 로직(`/dispatch-logic`)** 에 연결합니다.

```
[콜카드 발생]
      │
      ▼
[22D 성향 벡터화] ──┐
                    ├─→ [코사인 75% + H3 25% 점수] ─→ [콜카드 → 기사 성향 매칭 순위]
[기사 선호 벡터]  ──┘
      │
      ▼
[배차 로직에 "이 콜에 궁합 좋은 기사" 우선순위 전달]
      │
      ▼
[궁합 높은 기사에게 우선 배차] ─→ [수락률 ↑] ─→ [Expired ↓]
```

## 4.2 기대 효과

| 항목 | 변화 방향 |
|---|---|
| **수락률** | 콜 성향에 맞는 기사 우선 노출 → **상승** |
| **Expired(미배차)** | 안 잡히고 사라지는 콜 → **감소** |
| **승객 경험** | 대기/취소 감소 → **개선** |
| **기사 만족** | 선호 콜 위주 배차 → **공차 감소·만족 상승** |

## 4.3 핵심 메시지 (발표 마무리용)

> 우리는 **"기사마다 좋아하는 콜이 다르다"** 는 현장의 직관을
> **22차원 성향 벡터 + 코사인 유사도 + H3 공간 점수**로 정량화했고,
> 이를 **매칭 스튜디오로 검증**한 뒤 **배차 로직에 연결**합니다.
> 결과적으로 **콜카드 → 기사 성향(MBTI) 매칭**으로
> **Expired를 줄이는 것**이 이 프로젝트의 목표이자 성과입니다.
