# AI Priority Dispatch Developer Meeting Agenda

작성일: 2026-06-15
목적: 기존 배차로직에 22D 코사인 유사도 기반 AI 우선순위 정렬을 붙이기 전 개발자와 확인할 질문 목록

## 회의 목표

기존 배차로직을 전면 변경하지 않고, 기존 후보 기사 목록을 만든 뒤 그 후보들을 22D 코사인 유사도 기준으로 재정렬할 수 있는 위치를 찾는다.

## 먼저 공유할 문서

1. `handoff/AI_PRIORITY_DISPATCH_DEV_BRIEF.md`
2. `handoff/AI_PRIORITY_DISPATCH_SPEC.md`
3. `handoff/AI_PRIORITY_DISPATCH_ACCEPTANCE_TEST.md`

## 핵심 설명

이번 요청은 새 배차 엔진을 만드는 것이 아니다.

기존 배차로직이 이미 만들고 있는 후보 기사 목록을 받아서, 콜카드 조건과 기사 성향의 코사인 유사도가 높은 순서로 정렬한 뒤 우선 발송하는 방식이다.

```text
기존 후보 기사 목록
-> 후보 기사별 driver_mbti 조회
-> 콜카드 22D 벡터 생성
-> 코사인 유사도 계산
-> 유사도 높은 순서로 발송 우선순위 부여
-> 실패 시 기존 fallback 유지
```

## 개발자에게 확인할 질문

### 1. 기존 후보 기사 생성 위치

- 호출이 발생했을 때 후보 기사 목록은 어느 함수/API/서비스에서 만들어지는가?
- 후보 목록에는 `driver_id`가 포함되어 있는가?
- `vehicle_id` 또는 차량번호도 같이 넘어오는가?
- 후보 목록은 거리순인가, 순차순인가, 내부 점수순인가?
- 후보 목록 생성 이후 발송 직전 후보 순서를 바꿀 수 있는 지점이 있는가?

### 2. 콜카드 정보 접근

- 배차 시점에 아래 콜카드 필드를 사용할 수 있는가?
  - `callcard_id`
  - `asp_id`
  - 호출 요청 시간
  - 출발지 좌표
  - 도착지 좌표
  - 예상거리
  - 예상요금
  - 유료콜 여부
  - 탄력/일반 여부
  - ETA
- 콜카드 생성 시점과 배차 후보 생성 시점 중 어느 단계에서 이 정보가 확정되는가?

### 3. 기사 벡터 연결

- 기존 후보 기사 `driver_id`가 `driver_mbti.driver_id`와 같은 키인가?
- 기사 ID가 다르면 매핑 테이블이 필요한가?
- 후보 기사 중 `driver_mbti`가 없는 기사는 어떻게 처리하는 것이 좋은가?
  - 기존 순서 유지
  - AI 정렬 하위 배치
  - AI 실험 대상 제외

### 4. AI 정렬 삽입 위치

- 후보 생성 후 발송 전 후보 순서를 재정렬할 수 있는가?
- 후보 정렬 결과를 기존 발송 로직이 그대로 사용할 수 있는가?
- 1명씩 순차 발송인지, Top N에게 동시에 또는 묶음으로 발송하는 구조인지?
- 기존 fallback 로직은 어느 조건에서 실행되는가?

### 5. 이벤트 저장

- 콜카드 발송, 수락, 거절, 무응답, 만료 이벤트는 현재 어디에 저장되는가?
- 이벤트에 `dispatch_id` 같은 배차 시도 ID가 있는가?
- 없다면 AI 실험용 `dispatch_id`를 추가할 수 있는가?
- 발송 순위와 코사인 유사도를 함께 저장할 수 있는가?

### 6. 실험 범위

- 처음에는 특정 ASP나 일부 시간대에서만 켤 수 있는가?
- 전체 운영 적용 전 A/B 테스트가 가능한가?
- AI 우선배차를 끄는 feature flag가 필요한가?
- 장애나 성능 문제가 생기면 즉시 기존 배차로 되돌릴 수 있는가?

### 7. 성능과 장애 처리

- 후보 기사 수는 평균/최대 몇 명인가?
- 후보별 `driver_mbti` 조회를 배차 시점에 해도 지연이 허용되는가?
- 미리 캐시해야 하는가?
- `driver_mbti` 조회 실패 시 기존 배차 순서로 fallback할 수 있는가?

## 개발자에게 요청할 1차 구현

1. 기존 배차 후보 리스트를 만드는 지점을 확인한다.
2. 후보 리스트에 포함된 `driver_id` 기준으로 `driver_mbti`를 조회한다.
3. 콜카드 정보로 22D 콜카드 벡터를 만든다.
4. 후보 기사별 코사인 유사도를 계산한다.
5. 후보 리스트를 유사도 내림차순으로 정렬한다.
6. 정렬된 순서대로 기존 발송 로직에 넘긴다.
7. 실패하면 기존 fallback을 그대로 사용한다.
8. 발송 순위, 코사인 유사도, 결과 이벤트를 저장한다.

## 회의에서 결정해야 할 것

- AI 정렬을 삽입할 정확한 코드 위치
- 후보 기사 ID와 `driver_mbti` 연결 키
- `driver_mbti` 없는 후보 처리 방식
- 실험 시작 ASP/지역
- feature flag 필요 여부
- 배차 이벤트 저장 위치
- A/B 테스트 기준

## 회의 후 산출물

- 기존 배차로직 내 AI 정렬 삽입 위치
- 개발 담당자와 예상 일정
- 필요한 추가 DB/API 변경 목록
- 첫 실험 범위
- 검수 기준 적용 계획
