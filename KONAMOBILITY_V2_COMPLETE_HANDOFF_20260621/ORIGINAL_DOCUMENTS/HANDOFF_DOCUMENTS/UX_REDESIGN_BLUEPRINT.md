# Happycall PMO UX Redesign Blueprint

작성일: 2026-06-18
대상: Happycall PMO AI 우선배차 워크벤치
목적: 기존 화면을 조금 꾸미는 수준이 아니라, 사용자가 한눈에 이해하고 개발자에게 전달 가능한 제품 구조로 재설계한다.

---

## 1. 리뉴얼 판단

현재 화면은 기능은 많이 들어갔지만, 사용자가 무엇을 먼저 봐야 하는지 명확하지 않다.

문제는 색상이나 카드 스타일이 아니라 정보 구조다.

현재 섞여 있는 역할:

- 데이터 적재 확인
- 콜카드/기사 팩터 설명
- AI 매칭 시뮬레이션
- 개발자용 배차로직 전달서
- 운영 대시보드

앞으로는 이 역할을 4개 화면으로 고정하고, 각 화면은 하나의 코어 메시지만 가진다.

---

## 2. 전체 제품 구조

상단 탭은 4개로 고정한다.

1. 적재현황
2. 팩터리스트
3. 시뮬레이터
4. 배차로직

별도 랜딩/메인 화면은 만들지 않는다. 사용자는 바로 업무 화면으로 들어와야 한다.

권장 기본 진입 화면:

- 운영자/검수자: `/ingest`
- 기획/시연/개발 전달: `/simulator`

현재는 `/ingest`를 기본 시작점으로 유지한다.

---

## 3. 공통 레이아웃 원칙

모든 화면은 같은 구조를 공유한다.

### 공통 구조

- 상단: 고정 탭 네비게이션
- 좌측: 현재 화면의 입력/선택/필터/상태
- 중앙: 그 화면의 가장 중요한 시각화 또는 핵심 결과
- 우측: 선택된 대상의 상세 정보 또는 설명
- 하단: 로그, 드릴다운, 보조 설명, 검증 결과

### 금지할 것

- 같은 KPI를 여러 곳에 반복 표시하지 않는다.
- 모든 JSON, 모든 컬럼, 모든 설명을 화면에 노출하지 않는다.
- 작은 보조 텍스트로 핵심 내용을 설명하지 않는다.
- 기능 버튼과 정보 배지를 같은 시각 언어로 만들지 않는다.
- 개발자가 보는 내부 용어를 사용자 첫 화면에 그대로 노출하지 않는다.

### 타이포그래피

- 본문 최소 20px 기준
- 핵심 숫자 44px 이상
- 페이지 핵심 문장 48px 이상
- 보조 설명은 18px 미만으로 내리지 않는다.
- 긴 문장은 카드 안에 가두지 말고 2줄 이내로 줄인다.

### 화면 밀도

- 한 화면에 모든 기능을 넣지 않는다.
- 첫 뷰포트에는 코어 액션과 코어 결과만 남긴다.
- 상세는 선택, 펼침, 드릴다운으로 내려보낸다.

## 4. 페이지별 코어 정의

## 4.1 적재현황

### 이 페이지의 목적

현재 AI 우선배차 검증에 필요한 데이터가 어느 날짜까지 준비되어 있는지 확인한다.

### 사용자가 제일 먼저 봐야 하는 것

- 호출데이터 적재 기간
- 앱미터데이터 적재 기간
- 콜카드 생성 상태
- 기사 로그/기사 벡터 생성 상태
- 매칭 계산 상태
- 누락 또는 실패 날짜

### 코어 문장

"AI 우선배차 검증에 필요한 데이터가 어디까지 준비됐는지 확인한다."

### 코어 시각화

날짜 범위 타임라인.

- 가로 날짜 라인
- 호출데이터 존재일
- 앱미터데이터 존재일
- 기사 벡터 계산 완료일
- 매칭 계산 완료일
- 실패/누락일

### 좌측 패널

- 데이터 범위 요약
- 호출데이터: 시작일, 종료일, 일수, 건수
- 앱미터데이터: 시작일, 종료일, 일수, 건수
- Supabase 연결 상태

### 중앙 패널

- 날짜별 적재 커버리지 타임라인
- 각 날짜는 상태 칩으로 표시
  - 완료
  - 일부 완료
  - 누락
  - 실패
  - 확인 중

### 우측 패널

- 자동 적재 방식 설명
- 폴더 감시 명령
- 운영 기준 설명
- 가장 최근 처리 결과

### 하단 패널

- 실패 날짜 목록
- 누락 컬럼 목록
- 마지막 실행 로그 요약

### 숨길 것

- 원본 JSON 전체
- 불필요한 내부 파일 경로 반복
- 같은 적재 상태의 중복 카드

## 4.2 팩터리스트

### 이 페이지의 목적

콜카드와 기사 벡터가 어떤 팩터로 만들어지고, 어떤 계산식으로 비교되는지 설명한다.

### 사용자가 제일 먼저 봐야 하는 것

- 콜카드 팩터 22개
- 기사 팩터 22개
- 두 벡터가 같은 순서와 같은 의미로 비교되는지
- 5축 표시가 무엇을 요약하는지
- 코사인 유사도가 무엇을 의미하는지

### 코어 문장

"콜카드 조건과 기사 누적 운행패턴을 같은 22개 팩터로 놓고 비교한다."

### 코어 시각화

콜카드 카드와 기사 능력치 카드의 비교 화면.

권장 방향:

- 좌측: 선택된 콜카드 팩터
- 중앙: 22D 벡터 매칭 보드
- 우측: 선택된 기사 팩터
- 중앙에는 두 벡터의 유사성을 레이더 또는 능력치 비교 형태로 표시

### 표시할 22D 팩터 그룹

1. 시간대
   - dawn
   - morning
   - daytime
   - night

2. 요일
   - mon
   - tue
   - wed
   - thu
   - fri
   - sat
   - sun

3. 거리
   - short
   - medium
   - long

4. 요금
   - low_fare
   - mid_fare
   - high_fare

5. 상품/콜 유형
   - paid
   - free
   - surge
   - normal

6. 픽업 접근성
   - near

### 5축 표시

현재 5축은 임시 표시 체계다.

- 픽업 접근성
- 시간대 적합도
- 요일 적합도
- 운행거리 적합도
- 수익·상품 적합도

출발지 친숙도, 목적지 선호도는 H3/OD 데이터가 더 연결된 뒤 별도 축으로 확장한다.

### 좌측 패널

- 콜카드 선택
- 콜카드 주요 조건
- 콜카드 22D 팩터 리스트

### 중앙 패널

- 콜카드 22D와 기사 22D 비교
- 코사인 유사도 설명
- 5축 레이더 또는 능력치 차트
- 선택 팩터 강조

### 우측 패널

- 기사 선택
- 기사 누적 데이터 일수
- 신뢰도
- 기사 22D 팩터 리스트

### 하단 패널

- 계산식 설명
- 팩터별 원천 컬럼
- 콜카드와 기사에서 같은 팩터가 어떻게 다르게 계산되는지

### 숨길 것

- 개발자용 내부 변수명을 첫 화면에 그대로 노출하지 않는다.
- 설명 텍스트를 길게 나열하지 않는다.
- 22개 전체를 작은 표 하나로만 보여주지 않는다.

## 4.3 시뮬레이터

### 이 페이지의 목적

콜카드 조건을 입력하거나 실제 콜카드를 선택했을 때, 누적 기사 데이터 중 어떤 기사가 왜 우선 추천되는지 보여준다.

### 사용자가 제일 먼저 봐야 하는 것

- 입력된 콜카드
- 후보 기사군
- 최종 추천 Top 10
- 1순위 추천 이유
- 성향 유사도와 공간 적합도의 분리
- 거리/ETA는 시뮬레이션 값이라는 표시

### 코어 문장

"콜 하나가 들어오면, 후보 기사군이 AI 점수 기준으로 재정렬된다."

### 코어 시각화

콜카드가 중앙에 있고, 주변 기사 후보들이 연결선으로 배치되는 매칭 필드.

인터랙션 방향:

1. 콜카드 선택 또는 조건 입력
2. 콜카드 22D가 그려짐
3. 기사 후보군이 등장
4. 유사도가 높은 기사 연결선이 굵어짐
5. Top 10이 오른쪽 또는 하단에서 정렬됨
6. 기사 선택 시 레이더/팩터/추천 이유가 동기화됨

### 좌측 패널

- 콜카드 선택
- 조건 입력 모드
- 출발/도착 좌표
- H3
- 거리, 요금, 상품, 시간대

### 중앙 패널

- 매칭 필드
- 콜카드 노드
- 기사 후보 노드
- 연결선 굵기 = 최종 추천점수
- 연결선 색상 = 성향/공간 균형
- 선택 기사 강조

### 우측 패널

- 선택 기사 상세
- 최종 추천점수
- 성향 유사도
- 공간 적합도
- 출발지 적합도
- 목적지 적합도
- 데이터 신뢰도
- 예상 수락률: 시뮬레이션 표시
- 픽업거리/ETA: 시뮬레이션 표시

### 하단 패널

- Top 10 후보 리스트
- Top 4 비교
- 워터폴 why
- 팩터 드릴다운
- 반경 확장 시나리오

### 중요한 기능 결정

시뮬레이터는 고정된 실제 콜카드만 보여주는 화면이 아니다.

반드시 두 가지 모드를 제공해야 한다.

1. 실제 콜카드 선택 모드
   - Supabase의 실제 callcard_mbti 행을 선택
   - 실제 22D, H3, OD 키를 사용

2. 조건 입력 모드
   - 사용자가 시간, 요일, 거리, 요금, 상품, 출발/도착 H3 또는 좌표를 조정
   - 조정된 콜카드 벡터로 기사군 재계산
   - 실제 저장 데이터와 구분해서 시뮬레이션 입력임을 표시

### 숨길 것

- 설명문을 화면 중앙에 길게 두지 않는다.
- 작은 텍스트로 중요한 경고를 숨기지 않는다.
- ETA/거리 시뮬레이션 값을 실제 운영값처럼 보이게 하지 않는다.

## 4.4 배차로직

### 이 페이지의 목적

개발자에게 AI 우선배차의 구현 방향과 데이터 흐름을 전달한다.

### 사용자가 제일 먼저 봐야 하는 것

- 기존 배차 엔진은 유지한다
- AI는 후보군 안에서 우선순위만 정렬한다
- 22D 코사인 유사도와 H3 공간 적합도를 분리한다
- 미수락 시 기존 fallback 흐름으로 돌아간다

### 코어 문장

"기존 배차 엔진을 바꾸지 않고, 후보 기사군 안에서 AI 우선발송 순서만 만든다."

### 현재 적용 상태

이 페이지는 1차 리디자인 완료.

적용된 구조:

- 좌측 sticky 목차
- 우측 본문
- FLOW 수직 타임라인
- HANDOFF 데이터 카드
- SCORE 2x2 카드
- Grab/Uber/DiDi reference 카드
- POLICY 결정 항목

### 다음 개선 방향

- 실제 API 계약과 연결되는 다이어그램 추가
- 후보 생성 → 필터 → 랭킹 → 발송 → fallback Mermaid 또는 자체 플로우차트 추가
- 개발자가 바로 가져갈 수 있는 request/response 예시 접힘 영역 추가

- 상태 필터
- 반경 확장
- ETA 우선 구조

## 6. 리뉴얼 구현 순서

### Phase 1. 제품 구조 고정

- 공통 상단 탭 유지
- 각 페이지의 코어 문장 확정
- 각 페이지의 좌/중앙/우/하단 구조 확정
- 중복 KPI 제거

### Phase 2. 적재현황 리뉴얼

목표:

- 업로드 화면이 아니라 데이터 준비 현황 화면으로 전환
- 날짜별 커버리지 타임라인 추가
- 호출/앱미터/기사벡터/매칭 상태를 한 화면에서 확인

우선 수정 파일:

- `app/ingest/page.tsx`
- 필요 시 상태 API 정리

### Phase 3. 팩터리스트 리뉴얼

목표:

- 22D 팩터와 계산식을 쉽게 보여줌
- 콜카드와 기사 팩터를 능력치 비교처럼 시각화
- 5축은 보조 요약으로만 사용

우선 수정 파일:

- `app/vectors/page.tsx`
- `lib/matching-display-axis.ts`
- 필요 시 팩터 설명 상수 파일 추가

### Phase 4. 시뮬레이터 리뉴얼

목표:

- 콜카드 입력/선택에 따라 기사 후보군이 재정렬되는 인터랙션 구현
- 중앙 매칭 필드 중심 구조
- Top 10, 선택 기사, 레이더, why, H3 상세 동기화

우선 수정 파일:

## 9. 최종 목표 화면 느낌

이 제품은 일반 관리자 페이지가 아니라 AI 배차 실험실이어야 한다.

느낌:

- 어둡고 선명함
- 숫자는 크고 단호함
- 중앙에는 항상 하나의 핵심 장면
- 보조 정보는 접히거나 주변부에 있음
- 사용자는 5초 안에 현재 화면의 목적을 이해해야 함
- 개발자는 1분 안에 구현 흐름을 이해해야 함

서비스 레퍼런스에서 가져올 태도:

- Grab: 비교와 설명 가능성
- Uber: 후보 생성과 배차 흐름
- DiDi: 기사 패턴과 장기 선호

하지만 결과물은 Happycall PMO의 데이터와 목적에 맞춘 독립적인 워크벤치여야 한다.

## 10. 다음 작업

다음 작업은 `적재현황`부터 시작한다.

이유:

- 사용자가 현재 가장 헷갈리는 화면이다.
- 자동 적재 방식으로 바뀌었기 때문에 기존 업로드 중심 UI는 맞지 않는다.
- 전체 데이터 신뢰도를 먼저 보여줘야 팩터리스트와 시뮬레이터도 설득력이 생긴다.

적재현황 리뉴얼의 첫 목표:

1. 호출데이터와 앱미터데이터가 몇일부터 몇일까지 적재되었는지 첫 화면에 크게 표시
2. 날짜별 커버리지 타임라인 구현
3. 콜카드/기사로그/기사벡터/매칭 계산 상태를 중복 없이 정리
4. 자동 적재는 명령 안내와 최근 실행 상태만 보조로 표시
5. 실패/누락 날짜를 별도 패널로 분리
